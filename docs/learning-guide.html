3. Multi-Agent Systems (CrewAI)

What is a multi-agent system?

Instead of one monolithic LLM call, you create specialized "agents" — each with a focused role, expertise, and goal. They work together (sequentially or in parallel) to solve complex tasks.

Agent Architecture in PitchLens

# Each agent has: role, goal, backstory, tools, LLM config
market_scorer = Agent(
    role="Market Opportunity Analyst",
    goal="Evaluate TAM/SAM/SOM, market timing, and growth potential",
    backstory="You are a senior VC analyst who has evaluated 500+ pitch decks...",
    llm=llm_config,
)

# A task binds an agent to specific work
score_task = Task(
    description="Score the market opportunity from 1-10...",
    agent=market_scorer,
    expected_output="JSON with score, reasoning, suggestions",
)

Why 4 Separate Agents?

ReasonExplanation
SpecializationEach agent has a focused prompt optimized for one dimension
Parallelism4 agents run simultaneously — 4x faster than sequential
Fault isolationIf one agent fails, others still produce results
TestabilityEach agent can be unit-tested independently

Key Lesson: Prompt Engineering for Agents

The quality of agent output depends heavily on:

  • Role definition — "You are a senior VC analyst" vs "You are an AI assistant"
  • Structured output — Tell the agent exactly what format to return (JSON schema)
  • Grounding — "Base your analysis ONLY on the provided deck content"
  • Constraints — "Score must be integer 1-10, reasoning must be 50-500 words"

4. Pipeline Orchestration Patterns

The Pipeline

PDF Upload
    → Extractor (sequential, must complete first)
        → 4 Scorers (parallel, with retry)
            → Verdict Aggregator (sequential, waits for all)
                → Scorecard (persisted)

Key Implementation: asyncio.as_completed()

async def _run_all_scorers(extracted_content, analysis_id):
    # Create tasks for parallel execution
    tasks = {}
    for scorer_name, scorer_fn in scorer_configs:
        task = asyncio.create_task(
            _run_scorer_with_retry(scorer_name, scorer_fn, extracted_content)
        )
        tasks[task] = scorer_name

    # Process results AS THEY COMPLETE (not in order!)
    scores = []
    for completed_task in asyncio.as_completed(tasks.keys()):
        scorer_name, score, error = await completed_task
        if score:
            scores.append(score)
            # Stream partial result to frontend immediately
            await ws_manager.emit_partial_result(analysis_id, score)
    
    return scores
Key lesson: asyncio.as_completed() lets you process parallel tasks in completion order (not creation order). This enables real-time streaming of results as each agent finishes.

Retry Pattern

async def _run_scorer_with_retry(scorer_name, scorer_fn, content):
    for attempt in range(2):  # 0 = first try, 1 = retry
        try:
            result = await asyncio.wait_for(
                scorer_fn(content),
                timeout=120  # seconds
            )
            return (scorer_name, result, None)
        except (asyncio.TimeoutError, Exception) as e:
            if attempt == 0:
                continue  # retry once
            return (scorer_name, None, str(e))  # give up

5. WebSocket Real-Time Streaming

Why WebSocket over Polling?

  • Server pushes events to client instantly (no 1-second polling delay)
  • Bidirectional — can be extended for real-time chat
  • Lower overhead than repeated HTTP requests

Backend: Event Broadcasting

class WebSocketManager:
    _connections: Dict[str, Set[WebSocket]] = {}
    
    async def emit_stage_change(self, analysis_id: str, stage: PipelineStage):
        """Broadcast stage change to all connected clients for this analysis."""
        event = WSEvent(event_type="stage_change", stage=stage, timestamp=now())
        await self._broadcast(analysis_id, event)
    
    async def emit_partial_result(self, analysis_id: str, data: dict):
        """Stream a scorer result as soon as it completes."""
        event = WSEvent(event_type="partial_result", data=data, timestamp=now())
        await self._broadcast(analysis_id, event)

Frontend: Custom Hook

function useAnalysisWebSocket(analysisId: string) {
    const [state, setState] = useState({ currentStage: null, partialResults: [] });
    
    useEffect(() => {
        const ws = new WebSocket(`ws://localhost:8002/ws/analysis/${analysisId}`);
        
        ws.onmessage = (event) => {
            const data = JSON.parse(event.data);
            switch (data.event_type) {
                case 'stage_change':
                    setState(prev => ({ ...prev, currentStage: data.stage }));
                    break;
                case 'partial_result':
                    setState(prev => ({
                        ...prev,
                        partialResults: [...prev.partialResults, data.data]
                    }));
                    break;
                case 'complete':
                    setState(prev => ({ ...prev, isComplete: true }));
                    break;
            }
        };

        return () => ws.close();
    }, [analysisId]);
    
    return state;
}

6. RAG (Retrieval-Augmented Generation)

What is RAG?

Instead of relying on the LLM's training data alone, you retrieve relevant documents, inject them as context, then generate an answer. This grounds the LLM in actual data.

RAG Pipeline

User Question
    → Embed the question (convert to vector)
        → Search pgvector for similar deck chunks (cosine similarity)
            → Retrieve top-K relevant chunks
                → Construct prompt: "Based on these deck sections: {chunks}, answer: {question}"
                    → LLM generates grounded answer
                        → Return answer + cited sections

Chunking Strategy

# Split deck content into ~500-token chunks with overlap
chunks = split_text(extracted_content, chunk_size=500, overlap=50)

# Store each chunk with its embedding
for i, chunk in enumerate(chunks):
    embedding = await get_embedding(chunk.text)  # 1536-dim vector
    await db.execute(
        insert(DeckEmbedding).values(
            deck_id=deck_id,
            chunk_text=chunk.text,
            chunk_index=i,
            section_category=chunk.category,
            embedding=embedding,
        )
    )

Retrieval Query

-- pgvector cosine similarity search
SELECT chunk_text, section_category,
       1 - (embedding <=> query_embedding) AS similarity
FROM deck_embeddings
WHERE deck_id = $1
ORDER BY embedding <=> query_embedding
LIMIT 5;
Key lesson: RAG quality depends on chunking strategy. Too small = loses context. Too large = dilutes relevance. 500 tokens with 50-token overlap is a good default.

7. JWT Authentication

Token Flow

Login → Server issues Access Token (15 min) + Refresh Token (7 days)
    → Client stores tokens
        → Every API request: Authorization: Bearer {access_token}
            → Token expires? Use refresh token to get new access token
                → Refresh token expires? User must re-login

Why Two Tokens?

TokenLifetimePurpose
Access Token15 minShort-lived, used for every request. If stolen, limited damage window.
Refresh Token7 daysLong-lived, only sent to /refresh endpoint. Stored more securely.

Implementation

import jwt
from datetime import datetime, timedelta

def create_access_token(user_id: str) -> str:
    payload = {
        "sub": user_id,
        "exp": datetime.utcnow() + timedelta(minutes=15),
        "iat": datetime.utcnow(),
    }
    return jwt.encode(payload, SECRET_KEY, algorithm="HS256")

def verify_token(token: str) -> dict:
    try:
        return jwt.decode(token, SECRET_KEY, algorithms=["HS256"])
    except jwt.ExpiredSignatureError:
        raise HTTPException(401, "Token expired")
    except jwt.InvalidTokenError:
        raise HTTPException(401, "Invalid token")

8. Security Patterns

Prompt Injection Guard

Users can try to manipulate AI agents by injecting instructions into their deck content or chat messages.

# Patterns to detect:
INJECTION_PATTERNS = [
    r"ignore (all )?(previous|above) instructions",
    r"you are now",
    r"system prompt",
    r"disregard",
    r"\[INST\]",
    r"</?(system|user|assistant)>",
]

def scan(text: str) -> ScanResult:
    for pattern in INJECTION_PATTERNS:
        if re.search(pattern, text, re.IGNORECASE):
            return ScanResult(allowed=False)  # BLOCK
    return ScanResult(allowed=True)

Fail-Closed Design

# If the guard itself crashes, we REJECT (not allow) the request
try:
    result = injection_scan(user_input)
    if not result.allowed:
        raise SecurityViolation()
except Exception:
    # Guard failed — reject the request rather than letting unscanned input through
    raise HTTPException(503, "Service unavailable")

Rate Limiting (Sliding Window)

# Track timestamps of requests per user
# Window: last 60 seconds for general, last 3600 seconds for analysis
user_requests = get_recent_requests(user_id, window=60)
if len(user_requests) >= 60:
    retry_after = calculate_retry_after(user_requests)
    raise HTTPException(429, headers={"Retry-After": str(retry_after)})

9. React + TypeScript Patterns

React Hooks Rules (The Bug You Hit)

NEVER put hooks after an early return. React counts hooks by position. If you call 5 hooks on render 1, then early-return before the 5th hook on render 2, React crashes with "Rendered fewer hooks than expected." 
// ❌ WRONG — useMemo after early return
function Component() {
    const [data] = useState(null);
    if (data) return <Done />;  // early return
    const value = useMemo(() => ..., []);  // 💥 CRASH — hook after return
}

// ✅ CORRECT — all hooks before any return
function Component() {
    const [data] = useState(null);
    const value = useMemo(() => ..., []);  // all hooks first
    if (data) return <Done />;  // then early returns
    return <Loading />;
}

11. Testing Strategy

Testing Pyramid

         ┌─────────────┐
         │   E2E Tests  │  ← Few, slow, full user flows
         ├─────────────┤
         │ Integration  │  ← API contracts, DB queries
         ├─────────────┤

         ├─────────────┤

Property-Based Testing (Hypothesis)

from hypothesis import given, strategies as st

@given(scores=st.lists(st.integers(min_value=1, max_value=10), min_size=1, max_size=4))
def test_overall_score_is_mean(scores):
    """For ANY set of valid scores, overall = rounded mean."""
    result = compute_overall_score(scores)
    expected = round(sum(scores) / len(scores))
    assert result == expected
    assert 1 <= result <= 10

14. Interview Q&A (Ready-Made Answers)

Questions an interviewer will likely ask about this project, with concise ready-to-speak answers.

Project Overview Questions

Q: What does PitchLens do in one line?

A: It's an AI platform that analyzes startup pitch deck PDFs and generates investor-grade scorecards across 4 dimensions — Market, Team, Business Model, and Competition — using parallel AI agents in under 60 seconds.

Q: What was the business problem you were solving?

A: Startups lack access to fast, affordable, investor-perspective feedback on pitch decks. Getting human expert feedback costs $500–2000 or takes days/weeks. PitchLens automates this with AI, delivering structured scorecards instantly.

Q: Why multi-agent instead of a single LLM call?

A: Three reasons: (1) Specialization — each agent has a focused prompt tuned for one dimension, producing better outputs than a generic "evaluate everything" prompt. (2) Parallelism — 4 agents run simultaneously, making it 4x faster. (3) Fault isolation — if one agent fails, the others still produce results, giving a partial scorecard rather than total failure.

Q: Walk me through the architecture.

A: User uploads a PDF → FastAPI backend stores it and kicks off a background pipeline → Extractor agent parses PDF into structured sections → 4 scoring agents run in parallel (with retry on failure) → Verdict Aggregator combines scores into a final scorecard → Results are persisted to PostgreSQL and streamed to the frontend in real-time via WebSocket. Users can then ask follow-up questions through a RAG-based chat grounded in the deck content.

Technical Deep-Dive Questions

Q: How do you handle real-time progress updates?

A: WebSocket connection established when analysis starts. The orchestrator emits events at each stage transition (extracting, scoring_market, etc.) and streams partial results as each scorer completes. I use asyncio.as_completed() to process parallel tasks in completion order, enabling real-time streaming. Frontend has a custom React hook that maintains connection with exponential-backoff reconnection (3 attempts max).

Q: How does the RAG chat work?

A: After extraction, deck content is chunked into ~500-token segments and stored as vector embeddings in pgvector (1536-dim). When a user asks a question, I embed the query, do cosine similarity search to find the top-5 relevant chunks, construct a prompt with those chunks as context, and ask the LLM to answer grounded in that context. The response includes citations of which deck sections were referenced.

Q: What happens if an agent fails?

A: Each scorer gets one retry with a 120-second timeout. If both attempts fail, that category is marked as "failed" and the pipeline continues to aggregation with the available scores. The Verdict Aggregator handles partial inputs — it computes the overall score as a mean of available scores and notes which categories are missing. This is a graceful degradation pattern.

Q: How do you handle security?

A: Four layers: (1) JWT authentication with short-lived access tokens (15 min) and refresh tokens (7 days). (2) Prompt injection guard that pattern-matches user inputs before they reach agents — it's fail-closed, meaning if the guard itself errors, the request is rejected. (3) Rate limiting — 10 analyses/hour per user, 60 API calls/minute. (4) Data isolation — users can only access their own evaluations, 403 responses don't reveal resource existence.

Q: Why async Python? What's the benefit?

A: The pipeline involves heavy I/O waits (LLM API calls take 5-30 seconds each). With async, the event loop handles these concurrently — while one agent waits for LLM response, others can process. Without async, 4 sequential agent calls = 80s. With async parallel = ~20s. Also, WebSocket management requires async for non-blocking bidirectional communication.

Q: Explain your database design choices.

A: PostgreSQL with three key patterns: (1) Relational tables for users, decks, analyses, scorecards — standard normalized schema with foreign keys. (2) JSONB columns for flexible data (suggestions arrays, full scorecard JSON for round-trip storage). (3) pgvector extension for embedding storage — keeps vectors collocated with relational data, avoiding a separate vector DB infrastructure.

Q: How did you handle the frontend state management for WebSocket?

A: Custom React hook (useAnalysisWebSocket) encapsulates all WebSocket logic — connection, message parsing, reconnection, cleanup. It returns reactive state (currentStage, partialResults, isComplete) that components consume. I hit a React hooks bug early on — had a useMemo after an early return, which violated hooks rules and crashed the app. Fixed by moving ALL hooks above any conditional returns.

Behavioral / Design Decision Questions

Q: What was the hardest technical challenge?

A: Getting the parallel agent orchestration right with real-time streaming. The challenge was: 4 agents finish at unpredictable times, each result needs to be streamed immediately to the frontend, failures need retries without blocking other agents, and the aggregator must wait for all to complete. I solved it with asyncio.create_task() + asyncio.as_completed() which processes tasks in completion order, letting me emit WebSocket events the instant each scorer finishes.

Q: What trade-offs did you make?

A: (1) In-memory rate limiting instead of Redis — simpler but doesn't work across multiple server instances. Acceptable for single-instance deployment. (2) PyPDF2 for PDF parsing — lightweight but can't handle scanned images (OCR). Chose speed of implementation over completeness. (3) Equal weighting for scoring dimensions — investors weight them differently, but equal weighting is a fair default without user research data.

Q: If you had more time, what would you improve?

A: (See Future Improvements section below.)

Q: How did you test this?

A: Three layers: (1) Unit tests for pure functions (score computation, color mapping, token validation). (2) Property-based tests with Hypothesis — e.g., "for ANY valid set of scores, the overall score equals the rounded mean." (3) Integration tests with mocked agents — test the full pipeline flow without hitting the real LLM. Agents are mocked at the module boundary using monkeypatch.

Curveball Questions

Q: What if someone uploads a malicious PDF?

A: Defense in depth: (1) File type validation (must be valid PDF). (2) Size limits (20MB, 50 pages). (3) After extraction, the injection guard scans ALL extracted text before it reaches scoring agents. If injection patterns are detected, the pipeline aborts. (4) Fail-closed — if the guard itself fails, the request is rejected.

Q: How would you scale this to 1000 concurrent users?

A: (1) Move rate limiting to Redis (shared across instances). (2) Run multiple FastAPI instances behind a load balancer. (3) Use a task queue (Celery/RQ) for pipeline execution instead of background tasks. (4) Cache scorecard results. (5) WebSocket connections via Redis pub/sub for multi-instance broadcasting. (6) Consider serverless for the scoring agents (each is stateless and independent).

Q: Why not use LangChain instead of CrewAI?

A: CrewAI provides agent composition with roles, goals, and backstories out of the box — it's purpose-built for multi-agent systems. LangChain is more general-purpose (chains, tools, memory). For a team of specialized agents working on a shared task, CrewAI's abstractions fit better. That said, LangGraph (LangChain's agent framework) would also work well and offers more control over state machines.
